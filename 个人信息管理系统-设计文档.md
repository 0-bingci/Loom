# 个人信息管理系统 — 设计文档

> 版本：v1（初始设计）
> 定位：长期自用、可持续演化的个人系统。第一版做待办 + 按日期呈现的 dashboard，并对外暴露 API。

---

## 1. 项目定位与设计原则

### 1.1 它是什么

一个长期自用的个人信息管理系统。第一版聚焦三件事：记录待办事项（含每日循环任务）、一个按日期呈现并推送的 dashboard、对外暴露 API 供其他形态调用。

### 1.2 长期愿景

以后会围绕个人成长陆续建一系列生态项目（如习惯追踪、阅读记录、健康、财务等）。本系统未来会成为一个**拉取式聚合者**式的"中间站"：由它去认识其他系统的 API 并主动拉取数据、汇聚成统一视图，其他系统不需要知道它的存在。

依赖方向单一（本系统 → 外部），因此不需要维护一份对外的稳定契约，只需在外部接口变化时改自己这边的适配。

### 1.3 现阶段定位

**现在它是一个孤立系统。** 长期愿景的作用只是"现在别把自己钉死在角落里"，而**不是**要现在就去建平台。生态是以后的事，第一版不为它建任何基础设施。

### 1.4 核心原则（含理由）

- **API 优先。** 核心只管数据和逻辑；输入与呈现方式（网页、桌宠、微信……）都是 API 边界之外、可随时替换的"皮"。核心能跑，皮随便换；核心不通，再漂亮的皮也没用。

- **模块化单体，不拆微服务、不搭插件框架。** 跨进程边界不会让边界更干净，只会让它变昂贵、因而更难改（两套部署、网络会失败、写入无法原子化）。而它换来的独立扩容，单人系统几乎用不上。更关键的是：现阶段边界一定是错的、需要挪的——进程内挪一次是一个 commit，服务间挪一次是一场迁移。

- **沿现实真实存在的缝去拆分。** 独立项目 = 独立服务（有各自的生命周期，这是客观存在的缝）。待办 CRUD 没有独立存在的理由，它就是核心的一部分，不该被拆出去制造一条现实并不要求的边界。

- **把第一个功能当成外部消费者来做。** 待办模块通过核心能力的接口来调用，就像未来的外部项目那样。这等于从第一天起就免费地给未来的接口做压力测试。

- **自有数据 vs 镜像数据要分清。** 自有数据（待办、笔记）是真源，完整增删改查；未来从外部拉来的是镜像，本地只读、带来源信息，本系统不试图成为它们的真源。

- **不过度设计。** 单人项目真正的杀手，从来不是"架构建少了"，而是"脚手架建太多、东西还没好用就先累垮弃坑"。能用很多年的架构，是让你快速做出有用东西、又一直容易改的那种。简单在这里不是偷懒，是最利于长期的选择。

## 2. 范围

### 2.1 第一版包含

- 待办事项（**一次性** + **循环**，如"每日要做"）的增删改查
- 按日期**计算**出来的 dashboard（当天任务 + 每日任务）
- 每分钟轮询的提醒引擎，到点生成通知
- 供消费者（dashboard 等）调用的 API

### 2.2 明确推迟

- **碎片记录 / 笔记** —— 纯 CRUD，紧随其后就能加
- **信息流 / 外部数据拉取** —— 走通用"源"机制，以后加
- **任何前端形态**（桌宠、微信等）—— 在 API 边界之外，现在不考虑
- 复杂鉴权、实时推送投递基础设施、消息队列、插件框架等

## 3. 架构总览

系统由四个部分组成，横跨"两个世界"：

- **被动世界**（请求驱动）：API 层 + 数据的增删改查。客户端来调用它才动。
- **主动世界**（时钟驱动）：提醒 worker，每分钟轮询，不管有没有人在看都自己运行。这是本系统区别于普通 CRUD 应用的心脏。

两个世界在 **`notifications` 表**交汇：主动世界往里写"该提醒你的事"，被动世界的客户端从这里统一拉取。

### 组件

| 组件 | 职责 | 说明 |
| --- | --- | --- |
| API 层 | 统一入口、鉴权、CRUD | 所有消费者唯一的入口 |
| 提醒 worker | 每分钟轮询，生成到点提醒 | 主动运行；状态全在数据库，重启不丢 |
| 数据存储 | 持久化 | 默认 SQLite（单文件、零运维） |
| Dashboard | 按日期呈现的消费者视图 | 视图是**算出来的**，不是直接读表 |

### 技术默认

- **数据库：SQLite。** 单人用足够，一个文件、零运维。
- **主键：ULID（或 UUID）。** 全局唯一、不透明，为未来跨系统引用留好路——自增整数在跨系统引用/同步时会很难受，而现在改成 ULID 几乎零成本。
- **时区：UTC+8。** 所有"今天"与 `remind_time` 均以此计算。

## 4. 数据模型

### 4.1 核心建模难点

一次性任务的"完成"可以记在任务本身上（做完就是做完）。但循环任务不行：今天做了、昨天做了是两件独立的事，**"完成"对循环任务是按天的**，没法记在那一条任务定义上。

因此把"任务定义"和"每天的状态"拆成两张表——这是整个模型里唯一需要想对的点。

### 4.2 `tasks`（任务定义）

一条定义，不管循环多少天，永远只有一行。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | ULID | 主键 |
| `title` | text | 任务内容 |
| `due_date` | date, 可空 | 一次性任务用；循环任务留空 |
| `recurrence` | text, 可空 | 循环任务用，如 `daily` / `weekly:MON,WED` |
| `start_date` | date, 可空 | 循环生效起始日；留空 = 不限起点 |
| `end_date` | date, 可空 | 循环生效截止日；留空 = 永远循环 |
| `remind_time` | time, 可空 | 几点提醒 |
| `archived` | bool | 定义是否仍生效 |
| `created_at` | timestamp | |

### 4.3 `task_log`（每日状态，稀疏表）

记"某任务在某天发生了什么"。按 `(task_id, date)` 唯一。没有对应行 = 那天还没做、也还没提醒过。一次性任务同理，只是一辈子只会有一行。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | ULID | 主键 |
| `task_id` | ULID | 关联 `tasks` |
| `date` | date | 这一行对应的日期 |
| `done` | bool | 当天是否完成 |
| `done_at` | timestamp, 可空 | |
| `notified` | bool | 当天是否已生成过提醒 |

> 唯一约束：`(task_id, date)`

### 4.4 `notifications`（待呈现的提醒）

两个世界的交汇点。所有"该提醒你的事"不管来源，都汇进这里；客户端只拉这一张表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | ULID | 主键 |
| `task_id` | ULID | 来源任务 |
| `status` | text | `pending` / `read` |
| `created_at` | timestamp | |
| `read_at` | timestamp, 可空 | |

### 4.5 未来：自有 vs 镜像

未来从外部系统拉来的数据是**镜像**：本地只读，字段上带 `来源`、`外部 ID`、`拉取时间`、以及能跳回原系统的链接。真源永远在原系统，本地只存"够渲染统一视图"的那一层。这个区分现在就在设计里立好，以后不会乱。

## 5. 核心行为

### 5.1 Dashboard 视图（算出来的）

**核心原则：一次性任务会一直追着你，直到做完；循环任务是"当天有效"的，错过就错过，不累积。**

今天的视图由两部分算出来：

- **一次性任务**：`due_date ≤ 今天` 且未完成。这样上周该做、没做完的任务会以"逾期"状态一直挂着，不会到期后被悄悄吞掉。
- **循环任务**：`recurrence` 星期规则匹配今天，**且** `start_date` 为空或 `今天 ≥ start_date`，**且** `end_date` 为空或 `今天 ≤ end_date`。某天没做**不会累积到后面**——明天是崭新的一条，昨天那条只是留在历史里未完成。到期后（`今天 > end_date`）自动不再出现，无需任何清理。

再左连 `task_log`（on `task_id` 和 日期）拿到每条的完成状态。

**它是算出来的，不预生成。** 循环任务的"未来"是无限的，没必要提前给每天生成任务行——这也正是循环任务改规则、设区间时都不用清理旧数据的原因：本来就没生成过未来的行，`task_log` 也不受影响。

### 5.2 提醒循环（每分钟）

worker 每分钟醒来，找出满足以下条件的任务：

1. 今天该出现（一次性到期 or 循环规则匹配今天，且在生效区间内）
2. `remind_time` 已过
3. `task_log` 里今天这条还没标 `notified`
4. **今天还没完成**（已完成的不再提醒）

对每条：生成一条 `notification`，并把 `task_log` 今天这行标成 `notified`。

**时区：** 所有"今天"和 `remind_time` 一律按 **UTC+8** 计算。worker 若跑在 UTC，务必先换算——否则"今天"会差一天、提醒会在错误的本地时间响。

**去重要点：** 一定要标 `notified`，否则下一分钟它又符合条件，会被同一件事反复轰炸。日期天然帮你按天去重——明天换了日期、还没有 log 行，自动又能提醒一次。

**为什么用轮询而不是给每条任务注册定时任务：** 轮询的状态全在数据库里，服务器重启完全不影响、逻辑也更简单；而内存里的定时任务一重启就全丢、提醒也丢了。对个人系统，这种"笨办法"反而最稳。

### 5.3 通知生命周期

`created`（pending）→ 客户端拉取查看 → `read`。

### 5.4 推送模型

第一版是**纯拉取**：客户端主动来问（`GET /notifications?status=pending`），系统只管把通知堆在表里等它来拿。桌宠"点一下就获取"天然就是这个模型。

未来微信这类没法主动来问的渠道，需要真推——到时候再加一个投递层：通知照常先写进表（拉取立刻可用），需要主动推的渠道再额外发出去。两者可以共存。

## 6. 未来扩展路线

沿着这副骨架往上挂，地基都不用动：

- **笔记 / 碎片记录** —— 第四张纯 CRUD 表。

- **信息流 / 外部聚合** —— 做成通用的"**源（source）**"概念：一个源知道怎么拉、怎么把返回映射成本地记录、多久拉一次。信息流（RSS 等）是第一个实例；未来接自己的健康、阅读系统，就只是"注册一个新的源"。

- **更多消费者** —— 桌宠、微信等，都是 API 之外的皮。

- **中间站 / hub 愿景** —— 真正值得沉淀为核心的，是两个每个成长项目都想复用、又都不该各自重造的能力：**提醒 / 投递服务** 和 **统一事件时间线**。现有的待办 / 笔记 / 信息流，本质上是它们的第一批消费者。当前的通知系统已经是"发生了一件事 → 也许要提醒你"的特例——把它想成"事件派生"是个小的概念转变，不是多一套基础设施。

- **何时才真的把某东西拆成独立服务** —— 满足以下任一：需要不同的运行时 / 语言、资源或稳定性需求截然不同（如重型爬虫不想拖垮 API）、或它有独立的生命周期甚至要分享给别人。待办 CRUD 一条都不沾；未来那些生态项目条条都沾——这正是"它们独立、而这个不独立"的原因。

---

## 7. 非目标（现在明确不做）

- 消息队列 / 消息中间件（一张数据库表当队列，能撑到真撑不住那天）
- 插件 / 动态加载框架（第二个项目先做成独立服务，摸清真实接入形状再说；插件接口没法用一个插件设计好）
- 微服务拆分（先做模块清晰的单体）
- 无 schema 的"万物皆通用实体"模型（会毁掉让系统可查询的那份清晰）
- 复杂鉴权、实时推送基础设施

---

## 附：一个可延后的取舍

**拉取模式的固有延迟：** 只有到下次轮询时，才知道外部发生了什么。做 dashboard、看时间线完全没问题；但假如未来某个外部系统需要"出事了立刻提醒你"，纯拉取就得靠高频轮询，或者到那时才让那个系统改成主动推。这是很后面的事，现在放心里即可。
